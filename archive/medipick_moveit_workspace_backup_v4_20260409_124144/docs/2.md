# MediPick MoveIt 从 0 到可规划可执行

这份文档是给 0 基础同学看的。

目标不是只告诉你“运行哪个命令”，而是让你知道：

- 这个工作区里每个东西在做什么
- 为什么前面会出现“能规划不能执行”“看不到动画”“控制器空参数”这些问题
- 以后你自己怎么查、怎么改、怎么继续接真机

## 1. 最终目标是什么

我们现在希望做到的是：

- 在 MoveIt 的 RViz 可视化界面中看到机器人模型
- 在 RViz 里拖动目标姿态
- 点击 `Plan`
- 点击 `Plan & Execute`
- 机器人模型在 RViz 里按轨迹执行

这和“只做一个演示动画”不一样。

这里的执行链路是：

- MoveIt 负责规划
- `ros2_control` 负责接收轨迹并驱动机器人状态更新
- `joint_state_broadcaster` 把关节状态发出来
- `robot_state_publisher` 根据关节状态更新 TF
- RViz 根据 TF 和机器人模型显示动作

## 2. 这个工作区里几个关键包分别干什么

- `src/medipick_simple3_description`
  机器人 URDF 描述，里面有 MoveIt/ros2_control 要读的机器人模型。

- `src/medipick_moveit_config`
  MoveIt 配置包。
  这里面有 SRDF、OMPL 配置、控制器配置、RViz 配置、MoveIt 启动文件。

- `src/medipick_planning_server`
  这是你后面要继续扩展成业务规划服务的地方。
  现在里面有 `planning_server.py` 和一些测试/演示脚本。

- `scripts`
  一键脚本入口。

## 3. 先理解 MoveIt 可执行链路

如果只是 `Plan`，MoveIt 只需要：

- 机器人模型
- 关节限制
- 运动学求解器
- 规划器

如果要 `Plan & Execute`，还必须再有：

- `controller_manager`
- `joint_trajectory_controller`
- `joint_state_broadcaster`
- 机器人 URDF 中的 `<ros2_control>` 定义
- MoveIt 里的控制器配置

所以你前面遇到的情况是正常的：

- “能看到目标 marker，但不动”
- “Plan 成功但 Execute 没动画”
- “Execute 按钮灰掉”

这些很多时候不是规划器坏了，而是执行链没有真正接通。

## 4. 这次真正修了什么

这次为了让 MoveIt RViz 里能规划并执行，关键改了下面几类东西。

### 4.1 切回系统 ROS 2 Humble

前面尝试过 conda 环境，但 `ros2_control` 这一层在 conda 里容易出现版本和运行时不一致的问题。

所以最终采用的是：

- 使用系统 `/opt/ros/humble`
- 用系统 apt 安装官方 `ros2_control` 和 `ros2_controllers`

已经安装的关键包有：

- `ros-humble-ros2-control`
- `ros-humble-ros2-controllers`
- `ros-humble-controller-manager`
- `ros-humble-joint-trajectory-controller`
- `ros-humble-joint-state-broadcaster`

相关环境脚本在：

- `scripts/common.sh`

现在脚本默认优先用系统 ROS，而不是 conda。

### 4.2 给机器人模型补上 ros2_control 接口

MoveIt 规划只是算轨迹。
要让轨迹执行，机器人模型必须告诉 `ros2_control`：

- 有哪些 joint
- 这些 joint 支持什么 command interface
- 这些 joint 支持什么 state interface

这部分在：

- `src/medipick_simple3_description/urdf/simple3_moveit.urdf`

这里面使用了 mock hardware，所以在没有真机驱动的时候，也能先在 RViz 里看到执行效果。

注意：

- 这不是“假动画脚本”
- 这是标准 `ros2_control` 的 mock hardware 方式
- 后面接真机时，换的是硬件插件，不是整套 MoveIt 逻辑

### 4.3 配置 ros2_control 控制器

控制器配置在：

- `src/medipick_moveit_config/config/ros2_controllers.yaml`

这里定义了：

- `joint_state_broadcaster`
- `mobile_arm_controller`
- `tool_controller`

其中 `mobile_arm_controller` 是最重要的，它负责接收 MoveIt 发来的 `FollowJointTrajectory` 轨迹。

### 4.4 配置 MoveIt 控制器映射

MoveIt 自己还要知道：

- 规划好的轨迹该发给哪个控制器
- 这个控制器控制哪些 joint

这部分在：

- `src/medipick_moveit_config/config/moveit_controllers.yaml`

这里把 `mobile_arm_controller` 设置成默认控制器。

### 4.5 修复 demo.launch.py 中真正卡执行的 bug

最关键的坑在这里：

- `src/medipick_moveit_config/launch/demo.launch.py`

前面一直执行不起来，不是因为 planner 本身有问题，而是因为这里把 `ros2_control_node` 显式命名成了：

- `controller_manager`

在 Humble 下，这会让 `spawner` 给控制器灌参数时出问题，直接表现为：

- `'joints' parameter is empty`
- `'command_interfaces' parameter is empty`

也就是说：

- 控制器被 load 了
- 但配置阶段没有真正拿到 YAML 里的 joint 列表和接口列表
- 所以 configure 失败

最终修法非常关键：

- 去掉 `ros2_control_node` 上显式写死的 `name="controller_manager"`

修完后，日志会变成：

- `Configured and activated mobile_arm_controller`
- `Configured and activated tool_controller`
- `Configured and activated joint_state_broadcaster`

这说明执行链真的通了。

### 4.6 解决“反复启动后状态很怪”的问题

另一个很坑的问题是进程残留。

如果上一轮的这些进程没清干净：

- `move_group`
- `ros2_control_node`
- `controller_manager/spawner`
- `robot_state_publisher`
- `rviz2`

下一次再启动时就可能出现：

- `Controller already loaded`
- 又接着报 `no controller with this name exists`
- RViz 行为很怪
- Execute 时完全不稳定

所以我在脚本里补了启动前清理逻辑：

- `scripts/common.sh`
- `scripts/run_moveit_demo.sh`

现在每次运行 `run_moveit_demo.sh`，都会先清掉旧的 MoveIt/控制器/RViz 残留进程。

## 5. 现在怎么正确启动

先编译：

```bash
bash scripts/build.sh
```

再启动：

```bash
bash scripts/run_moveit_demo.sh
```

这个脚本现在会自动做两件事：

- 启动前清理旧进程
- 用系统 `/opt/ros/humble` 环境启动

## 6. 打开 RViz 后怎么操作

### 6.1 第一步先确认界面正常

你应该能看到：

- 机器人模型
- MoveIt 的 `MotionPlanning` 面板
- 末端交互 marker

### 6.2 第二步先选对规划组

优先用：

- `Planning Group = mobile_arm`

不要默认一直用：

- `arm`

原因很简单：

- `arm` 只会动机械臂
- `mobile_arm` 才会把底盘和机械臂一起纳入规划自由度

### 6.3 第三步先做小动作验证

第一次不要把目标拖太远。

建议：

- 在当前末端附近拖一点点
- 先验证 `Plan`
- 再验证 `Plan & Execute`

如果一上来就拖到很高、很远、姿态变化很激进的位置，你很容易把“模型可达性问题”和“执行链问题”混在一起。

## 7. 如何看日志判断是哪一层坏了

这是最重要的排错思路。

### 7.1 如果是规划失败

常见日志：

- `Unable to solve the planning problem`
- `NO_IK_SOLUTION`

这说明是：

- 目标不可达
- 碰撞约束太紧
- 逆解找不到
- 起始状态不合法

这时候先看规划问题，不要先怀疑控制器。

### 7.2 如果是控制器没起来

常见日志：

- `'joints' parameter is empty`
- `'command_interfaces' parameter is empty`
- `Failed to configure controller`

这说明是：

- `ros2_control` 控制器参数没有真正加载成功

### 7.3 如果是旧进程冲突

常见日志：

- `Controller already loaded`
- `Could not configure controller with name 'xxx' because no controller with this name exists`

这往往说明：

- 旧的 `controller_manager` 或 spawner 还在
- 新老两套进程打架了

### 7.4 如果是执行链没接通

常见现象：

- `Plan` 能成功
- `Plan & Execute` 没动画
- 或者 Execute 按钮状态不对

这时候重点查：

- `ros2_control_node` 是否起来
- `mobile_arm_controller` 是否 `configured and activated`
- `joint_state_broadcaster` 是否起来
- MoveIt 的 controller 配置是否匹配

## 8. 现在还有哪些东西没做

现在已经做到：

- MoveIt RViz 中规划
- MoveIt RViz 中执行
- 控制器链路打通

但下面这些还没有完全做完：

- 真实点云接入 MoveIt planning scene
- 真机硬件驱动接入 `ros2_control`
- 抓取目标自动从视觉结果生成
- 货架和药盒的真实环境更新
- SRDF 中 `virtual joint` 语义完善
- Octomap 3D 传感器插件配置

所以现在这套更准确地说是：

- “MoveIt + 标准 ros2_control mock hardware 的可规划可执行验证链”

它已经非常适合：

- 调模型
- 调关节组
- 调规划参数
- 调控制器配置
- 验证执行链是否通

## 9. 你以后继续接真机时怎么想

后面接真机时，不要把现在这套推翻重来。

更合理的思路是：

- 保留 MoveIt 配置
- 保留 MoveIt 控制器映射
- 保留规划服务接口
- 把 mock hardware 换成真实硬件插件
- 把静态/随机障碍输入换成真实点云或障碍建模输入

也就是说，后面主要替换的是：

- 硬件层
- 感知层

而不是：

- MoveIt 规划层
- 任务调度层

## 10. 这次最重要的经验总结

如果以后你再遇到“MoveIt 里能看不能动”，优先按这个顺序查：

1. 先看是不是 `Plan` 本身失败了。
2. 再看 `ros2_control_node` 有没有起来。
3. 再看 `mobile_arm_controller` 有没有真正 `configured and activated`。
4. 再看 `joint_state_broadcaster` 有没有起来。
5. 再看是不是旧进程冲突。
6. 最后再去怀疑规划器参数。

还有一个很重要的经验：

- 很多看起来像“规划器不行”的问题，实际根因在执行链、控制器配置、模型接口、或者进程残留。

## 12. 最后给 0 基础同学的一句话

你可以把这套东西先粗暴地理解成三层：

- URDF/SRDF 决定“机器人是什么”
- MoveIt 决定“怎么规划过去”
- ros2_control 决定“怎么算出来的轨迹真正执行出来”

只要你一直按这三层去想，遇到问题就不会乱。
