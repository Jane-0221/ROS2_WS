# 当前版本问题梳理与简化重构方案

这份文档不是功能说明，而是给当前版本“停下来做一次结构诊断”用的。

目标很明确：

- 机器人去货架隔层中间
- 吸盘姿态对准目标位姿
- 先确保底盘和手臂处于安全、可解释的姿态
- 再完成一次短距离、可控的吸取动作

当前版本之所以越来越难调，不是因为某一个参数错了，而是因为**流程里同时存在太多互相竞争的预设和 fallback**，导致：

- 任务虽然“有时能跑完”
- 但动作逻辑不清晰
- 很难判断到底是哪条分支真正起作用
- 一条分支修好后，另一条 fallback 又把旧问题带回来


## 1. 当前版本的主要问题

### 1. `OBSERVE` 阶段做了太多事

现在 `OBSERVE` 里同时在做：

- 计算 `prepare_pose`
- 计算 `retreat_pose`
- 搜索动态 `prepare candidate`
- 推出 `base_goal_pose`
- 推出 `target_lift_height`
- 有时还为后续 `final_approach` 提前缓存可行结果

这会带来一个问题：

- 底盘其实还没动
- 升降杆也还没到位
- 但系统已经在用“未来的几何状态”猜后面的姿态和轨迹

结果就是：

- 观察阶段选出来的 `prepare_candidate`
- 到真正执行 `BASE_APPROACH / LIFT_ALIGN` 后
- 已经不再是最合适的候选


### 2. 当前版本有多套 `prepare` 逻辑同时存在

现在实际上并不是只有一个“预备姿态求解器”，而是至少有这些入口：

- 动态 branch candidate 搜索
- template candidate 搜索
- deterministic `prepare_seed`
- `prepare_pose` planning retry
- `prepare_use_planning` 这条可选 refine 分支

这会导致：

- 你看到的最终动作，不一定来自当前以为在调的那条逻辑
- 一条路径被修掉后，另一条 fallback 仍可能生成奇怪姿态


### 3. 当前版本有多套 `final_approach` 逻辑同时存在

现在 `FINAL_APPROACH` 也不是单一路径，而是可能走：

- prefetched direct final
- candidate 内缓存的 final result
- seeded IK local insertion
- Cartesian path
- OMPL pose planning

这会导致：

- 明明以为在测试“直插”
- 实际却可能落到“局部 IK / fallback / 旧 branch”

用户看到的现象就是：

- 有时候绕大圈
- 有时候姿态很怪
- 有时候明明没碰到目标却看起来还在往下走


### 4. 小车对准逻辑和手臂姿态逻辑耦合得太早

用户当前看到的根因之一是：

- 小车移动时目标点接近机器人中心
- 手臂又不是严格保持“收臂安全姿态”
- 所以底盘一动，吸盘就可能已经伸进货架里

这说明：

- `BASE_ALIGN` 不应该依赖“未来的 prepare 手臂姿态”
- `BASE_ALIGN` 必须只在“安全收臂包络”下求底盘站位


### 5. `PREPARE_POSE` 的本质定义还不够收敛

当前代码里 `prepare_pose` 虽然名义上是：

- 与目标同姿态
- 沿法向后退一点

但实际执行时又混入了：

- 大量预设 arm 模板
- 各种 branch seed
- 各种 geometry score

结果是：

- `prepare_pose` 变成了“很多启发式的混合物”
- 而不是一个明确的动作定义


## 2. 当前版本最值得砍掉的复杂性

下面这些东西不是完全不能存在，但它们不应该同时在线：

### 建议去掉或暂时禁用

- `OBSERVE` 阶段里的动态 `prepare candidate` 全量搜索
- template candidate fallback
- `prepare_pose` 的 deterministic seed + refine 双层逻辑
- `FINAL_APPROACH` 的多套 fallback 并存
- 在 `OBSERVE` 阶段预缓存 `final_result`

### 建议保留但收成单一路径

- 一个明确的 `STOW_SAFE` 姿态模板
- 一个明确的 `BASE_ALIGN` 计算器
- 一个明确的 `LIFT_ALIGN` 高度计算器
- 一个明确的 `PREPARE_POSE` 定义
- 一个明确的 `FINAL_APPROACH` 规划器


## 3. 简化后的主流程

建议改成下面这 4 段主流程。

当前准备实施的新版默认配置就是按这个思路收的：

- `use_simplified_pipeline = true`
- `prepare_group_name = arm`
- `final_group_name = arm_no_lift`
- `retreat_group_name = arm_no_lift`
- `base_standoff ≈ 0.68`
- `base_lateral_offset ≈ 0.18`
- `final_prefer_seeded_ik = false`

也就是说：

- `PREPARE` 允许升降杆参与几何求解
- `FINAL/RETREAT` 强制只用手臂
- `BASE_ALIGN` 默认让目标偏向机器人右侧，而不是落在机器人中心线上

### 1. `STOW_SAFE`

目的：

- 手臂收回到安全包络内
- 底盘移动时，吸盘不能探进货架

要求：

- 只允许 `raise_joint` 和 `r1..r6` 到一个固定安全模板
- 不在这个阶段猜目标姿态
- 只做“收起来”

当前用户认可的一组近似安全姿态可作为起点：

- `raise_joint ≈ 0.22`
- `r1 ≈ -41°`
- `r2 ≈ 68°`
- `r3 ≈ -261°`
- `r4 ≈ 114°`
- `r5 ≈ 26°`
- `r6 ≈ 9°`

后续要做的是：

- 用 FK 验证这组姿态下，吸盘是否完全在机器人安全包络里
- 把它固定成“底盘移动专用收臂姿态”


### 2. `BASE_ALIGN`

目的：

- 只靠小车对准
- 不让目标落在机器人中心正下方
- 不让手臂在底盘移动阶段参与“帮忙对准”

要求：

- 只求底盘位姿
- 手臂保持 `STOW_SAFE`
- 底盘目标必须满足：
  - 面向货架开口
  - 目标相对底盘不落在“太靠中心/太靠身体”的区域
  - 预留一段安全前向距离

这一步的关键不是“让吸盘已经接近目标”，而是：

- 给下一步的升降和手臂工作区创造条件


### 3. `LIFT_ALIGN`

目的：

- 让升降杆把末端高度带到目标附近
- 但这一步只负责高度，不负责横向和前向对准

要求：

- 只动升降杆
- 手臂仍保持 `STOW_SAFE`
- 只把高度带到“可工作带”


### 4. `PICK`

这里再拆成两个子阶段，但逻辑属于同一段：

#### 4.1 先尝试 `DIRECT_PICK`

思路：

- 在 `STOW_SAFE + BASE_ALIGN + LIFT_ALIGN` 之后
- 先试一下能不能直接完成一条短距离 final approach

如果能：

- 直接做 `FINAL_APPROACH`
- 不再进入 `PREPARE_POSE`

#### 4.2 只有直捡不行，才进入 `PREPARE_POSE`

这时 `PREPARE_POSE` 的定义必须非常简单：

- 与目标姿态相同
- 在目标法向上后退 `d_prepare`
- 在垂直于法向的平面上投影重合

而且这一步不应该再有很多 arm 模板混进来。

真正要做的是：

- 先确定底盘、升降杆已经给出了合理工作区
- 然后只求一个“右臂能无碰撞到达的预备姿态”

最后：

- `FINAL_APPROACH` 只保留一条逻辑
- 最好是短距离、姿态基本固定、只向前插入


## 4. 重新定义“是否成功”

当前系统里最危险的问题之一是：

- 轨迹执行成功
- 但末端没真正碰到目标
- 仍然可能继续往下走

所以新的成功标准必须统一成：

### `PREPARE_POSE` 成功

- 末端姿态与 `prepare_pose` 误差在阈值内
- 轨迹全程不碰撞

### `FINAL_APPROACH` 成功

- 末端姿态与 `target_pose` 误差在阈值内
- 轨迹全程不碰撞
- 到达点不是“看起来差不多”，而是 FK 明确验证

### `COMPLETED`

只能在满足以下条件后进入：

- `FINAL_APPROACH` 成功
- 如果有回撤，`RETREAT` 成功


## 5. 下一轮重构建议

不要继续在当前版本上叠更多 heuristic。

建议下一轮按下面顺序重构：

1. 暂时冻结当前动态 `prepare candidate` 搜索、template fallback、prefetched final 逻辑。
2. 固化 `STOW_SAFE` 为一组明确可验证的安全姿态。
3. 单独重写 `BASE_ALIGN`：
   - 目标不落机器人中心
   - 手臂保持安全收拢
4. 单独重写 `LIFT_ALIGN`
5. `PICK` 阶段只保留：
   - `DIRECT_PICK`
   - `PREPARE_POSE -> FINAL_APPROACH`
6. `FINAL_APPROACH` 只保留一条主路径，不再多套 fallback 并行


## 6. 结论

当前主要不是“规划器参数还不够好”，而是：

- 底盘对准、升降、预备姿态、最终插入
- 这几件事的职责边界还不够清楚

只要继续把：

- 多套 prepare 逻辑
- 多套 final 逻辑
- 观察阶段过早猜后续状态

这些复杂性一起留着，后面还是会反复出现：

- 手臂穿货架
- 小车走位不合理
- prepare 很怪
- final 绕圈
- 看起来完成但其实没对准目标

所以接下来最对的方向不是继续叠补丁，而是：

- 先把流程收成更少、更清晰的 4 段主链
- 再逐段验证
