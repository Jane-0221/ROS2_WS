# 面向真机部署的货架吸取方案

这份文档不是在讲“怎么做一个 RViz 演示”，而是在讲一条以后可以落到真机上的主链。

你的目标可以概括成：

1. 机器人正面对着货柜
2. 相机获取点云
3. 在货架两个隔层之间生成一个目标点和目标位姿
4. 机器人分阶段完成靠近、对齐、抬升、摆臂、末端接近、最终吸取
5. 先在实时仿真里验证，再迁移到真机

这条链里，真正要做的是：

- 感知和目标生成
- 任务状态机
- 分阶段运动规划
- 与底盘导航的接口
- 仿真验证

不是单独做一个 demo 脚本。

## 1. 推荐的总体架构

建议把系统拆成 5 层。

### 1.1 感知层

职责：

- 接收相机点云
- 做点云预处理
- 建立货架局部环境模型
- 找出“两个隔层之间”的目标点和目标法向

输入：

- `/camera/depth/points` 或等价 `PointCloud2`

输出：

- 过滤后的环境点云
- 货架层板模型
- 目标点
- 目标姿态

### 1.2 场景建模层

职责：

- 把感知结果转换成 MoveIt 规划场景
- 维护货架、障碍、目标区域

输入：

- 感知层输出的结构信息

输出：

- MoveIt planning scene

### 1.3 任务编排层

职责：

- 把“看见目标”变成一个多阶段任务
- 管理每一阶段是否成功
- 决定什么时候调导航，什么时候调 MoveIt

这层才是你的主算法骨架。

### 1.4 运动规划层

职责：

- 接收每一阶段的目标位姿或目标关节条件
- 调用 MoveIt 规划
- 检查碰撞、可达性、姿态约束

这层可以继续复用你现在的 [planning_server.py](/home/lzoolloozl/projects/medipick_moveit_workspace/src/medipick_planning_server/scripts/planning_server.py)。

### 1.5 执行层

职责：

- 底盘控制或导航执行
- 机械臂控制执行
- 反馈当前位置和状态

这里要和真机控制器保持接口一致。

## 2. 推荐的任务状态机

你现在要的动作流程，建议拆成下面 7 个状态。

### 2.1 `OBSERVE`

目标：

- 机器人面向货架
- 头部相机扫描或固定观察
- 获取点云

动作：

- 头部 `h1/h2` 先摆到观察姿态
- 等待点云稳定 0.5 到 1 秒

输出：

- 货架层板
- 隔层入口
- 目标点和目标位姿

### 2.2 `STOW_ARM`

目标：

- 手臂先收拢
- 小臂靠向大臂
- 保证底盘靠近时机械臂不容易碰货架

这一步应该是一个固定的“收纳姿态”。

推荐做法：

- 定义一个 `stow` 或 `compact_ready` 关节姿态
- 只要进入移动底盘阶段，就先回到这个姿态

### 2.3 `BASE_APPROACH`

目标：

- 底盘移动到目标前方一个偏置位
- 不要让底盘正中心严格对准目标
- 给右臂留空间

这里不是导航算法本身，而是给导航一个局部目标位姿。

你应该输出：

- `base_goal_position`
- `base_goal_yaw`

这个目标位姿由目标点反推。

推荐规则：

- 在目标点前方保留一个 standoff 距离
- 横向保留一个偏置，让右臂工作空间更舒服
- 朝向目标点，但允许存在一个小偏角

例如：

- 前向距离：`0.45 ~ 0.65 m`
- 侧向偏置：`0.10 ~ 0.20 m`
- 朝向偏角：`5° ~ 15°`

### 2.4 `PREPARE_POSE`

目标：

- 底盘到位以后
- 升降杆和右臂摆到“货架入口前的准备姿态”
- 末端高度和目标点基本平齐

这一步不要直接插进货架内部。

应该先到一个准备位姿：

- 位于层板入口外
- 姿态已经和吸取方向基本一致
- 高度接近目标高度

这一步可以允许底盘不动，只动：

- `raise_joint`
- `r1~r6`

如果以后你发现光靠右臂不够，也可以让 `mobile_arm` 在这一步做微调。

### 2.5 `FINAL_APPROACH`

目标：

- 从准备姿态向目标吸取姿态推进
- 尽量只动右臂

这是最关键的一段。

建议：

- 起点是准备姿态
- 终点是吸取姿态
- 约束姿态基本不变
- 主要让末端沿货架法向前进

也就是说：

- `pregrasp` 到 `grasp`
- 末端轨迹尽量短
- 尽量减少底盘动作

### 2.6 `SUCTION`

目标：

- 到位后触发吸取

当前先不做真实药盒时，可以只做：

- 到达目标位姿后等待
- 标记任务成功

真机时再接：

- 真空泵控制
- 压力传感器反馈

### 2.7 `RETREAT`

目标：

- 吸取完成后退出货架

建议：

- 先后退
- 再抬升或回中

不要直接大幅度摆臂，不然容易碰层板。

## 3. 目标点和目标姿态怎么生成

你说的目标不是药盒，而是“两个隔层之间的一个目标点和目标位姿”。

这非常适合作为第一版。

### 3.1 输入

从点云中识别：

- 上层板平面
- 下层板平面
- 货架前沿
- 货架背板

### 3.2 目标点

目标点可以定义成：

- 两层板中间高度
- 某个货格的中心线
- 距离背板前方一定距离的位置

例如：

```text
target_z = (upper_shelf_bottom_z + lower_shelf_top_z) / 2
target_x = shelf_inner_depth_center 或者 shelf_back_x - margin
target_y = 货格中心
```

### 3.3 目标姿态

如果吸盘从正前方进入货架：

- 吸取方向沿货架深度轴
- 吸盘法向朝向货架内部

你现在的 MoveIt 里常用的末端朝向就可以继续保持：

- 让 `sucker` 朝向货架内部

这一步关键不是“姿态多复杂”，而是：

- 姿态定义要稳定
- 与末端工具坐标系一致

## 4. 底盘目标怎么从目标点反推

这一步不要塞到 MoveIt 里做。

应该由任务编排层单独算一个底盘目标。

### 4.1 基本思路

假设目标点是 `p_target`。

先估计货架法向 `n_shelf`，也就是“从机器人朝货架看过去的进入方向”。

然后构造：

```text
base_goal = p_target - standoff * n_shelf + lateral_offset * t_shelf
```

其中：

- `standoff` 是前后距离
- `t_shelf` 是货架横向方向
- `lateral_offset` 是给右臂预留的侧向偏置

### 4.2 机器人的朝向

朝向不要严格对着目标点正中心。

建议：

- 朝向货架法向附近
- 再加一个小偏角

原因：

- 避免底盘中心正好挡在目标前
- 给右臂的工作空间留更顺的入射角

## 5. 规划时每一阶段分别用什么 group

### 5.1 观察阶段

用：

- `head_controller`

不需要 MoveIt 复杂规划，先用简单关节控制就够。

### 5.2 底盘靠近阶段

不要让 MoveIt 代替导航。

用：

- 任务层输出局部底盘目标
- 交给导航/底盘控制模块执行

### 5.3 准备姿态阶段

建议优先用：

- `mobile_arm` 或者 `arm`

更推荐第一版先这样：

- 底盘到位以后
- `arm` + `raise_joint` 完成准备姿态

### 5.4 最终接近阶段

建议用：

- `arm`

这一步尽量只让机械臂进货架，避免底盘微抖影响精度。

## 7. 仿真里怎么做“实时运算看效果”

既然你现在准备走 Gazebo，这里建议直接做一套固定仿真场景。

### 7.1 场景内容

至少包括：

- 机器人
- 多层货架
- 若干层板
- 一个“目标货格”
- 点云相机

### 7.2 实时链路

仿真里应该跑这条链：

```text
Gazebo 相机
-> PointCloud2
-> shelf_scene_extractor
-> 目标点/目标姿态
-> pick_task_manager
-> 底盘局部目标
-> MoveIt 准备姿态规划
-> MoveIt 最终接近规划
-> Gazebo 执行
```

### 7.3 第一版不要追求太复杂

先做这几个简化：

- 货架静态
- 目标货格固定
- 不做真正药盒识别
- 只生成货格中点和姿态

这已经足够验证主链。

## 11. 当前已经先落下来的任务状态机接口

现在仓库里已经先放了一版状态机骨架：

- `src/medipick_planning_server/scripts/pick_task_manager.py`

这一版的定位不是把没完成的模块写死，而是先把主链接口定出来。

### 11.1 输入接口

- `/medipick/task/target_pose`  
  类型：`geometry_msgs/PoseStamped`  
  含义：感知层给出的目标点和目标姿态。当前没有真实视觉时，可以直接由仿真或测试节点发布。

- `/joint_states`  
  类型：`sensor_msgs/JointState`  
  含义：当前机器人真实关节状态，用来作为准备姿态阶段的规划起点。

### 11.2 控制接口

- `/medipick/task/start`  
  类型：`std_srvs/Trigger`  
  含义：收到后开始一次完整的 6 阶段任务。

- `/medipick/task/reset`  
  类型：`std_srvs/Trigger`  
  含义：清空当前任务状态，回到 `IDLE`。

- `/medipick/task/mark_base_arrived`  
  类型：`std_srvs/Trigger`  
  含义：底盘模块告诉任务层“我已经到位了”，状态机才会继续往下走。

### 11.3 输出接口

- `/medipick/task/stage`  
  当前任务阶段

- `/medipick/task/event`  
  状态机事件日志

- `/medipick/task/base_goal`  
  任务层根据目标点反推出的底盘局部目标位姿

- `/medipick/task/prepare_pose`
- `/medipick/task/final_pose`
- `/medipick/task/retreat_pose`  
  三个关键阶段目标位姿

- `/medipick/task/stow_joint_state`  
  收臂阶段的目标关节姿态接口

- `/medipick/task/planned_trajectory`  
  每一段 MoveIt 规划返回的轨迹

### 11.4 当前 6 个阶段

当前这版状态机固定为：

1. `OBSERVE`
2. `STOW_ARM`
3. `BASE_APPROACH`
4. `PREPARE_POSE`
5. `FINAL_APPROACH`
6. `RETREAT`

说明：

- `OBSERVE` 阶段等待一个新鲜的 `target_pose`
- `STOW_ARM` 阶段先发布收臂姿态
- `BASE_APPROACH` 阶段只输出底盘目标，不替代导航
- `PREPARE_POSE` / `FINAL_APPROACH` / `RETREAT` 会真实调用 `planning_server`

### 11.5 现在为什么这样设计

因为当前工程里这些部分还没完全接好：

- SLAM / 导航
- 真正的视觉检测
- 点云建图后的货架结构提取
- 真空吸附执行逻辑

所以这版状态机先把真正应该稳定存在的主链固定下来：

```text
感知接口
-> 状态机
-> 底盘目标接口
-> MoveIt 两段或三段规划
-> 执行接口
```

这样后面你把任何一个未完成模块接进来，都不用再推翻任务层结构。

### 11.6 现在怎么直接测这条状态机

启动：

```bash
bash scripts/run_pick_task_manager.sh auto_accept_base_arrival:=true
```

手动输入一个目标位姿：

```bash
source /opt/ros/humble/setup.bash
source install/setup.bash

ros2 topic pub --once /medipick/task/target_pose geometry_msgs/msg/PoseStamped \
'{
  header: {frame_id: world},
  pose: {
    position: {x: 0.78, y: -0.05, z: 1.08},
    orientation: {x: 0.0, y: 0.7071068, z: 0.0, w: 0.7071068}
  }
}'
```

触发任务开始：

```bash
ros2 service call /medipick/task/start std_srvs/srv/Trigger "{}"
```

如果你不想自动通过底盘阶段，就把 `auto_accept_base_arrival` 设成 `false`，然后等底盘模块或者你手动确认时再调用：

```bash
ros2 service call /medipick/task/mark_base_arrived std_srvs/srv/Trigger "{}"
```

查看当前状态：

```bash
ros2 topic echo /medipick/task/stage
ros2 topic echo /medipick/task/event
```
