# 基于离散升降杆的新版拾取分段方案

这份文档解决的是一个已经明确暴露出来的问题：

- 真实机器人的 `raise_joint` 不是一个可以和底盘、右臂一起做连续时间轨迹跟踪的自由度
- 它更像一个“单独到某个高度”的执行机构
- 所以当前“底盘 + 升降杆 + 手臂一起规划”的思路，不能直接作为真机部署方案

结论先写在前面：

- 后续主方案不再把升降杆当成连续联合规划轴
- 升降杆只在阶段边界动作
- 联合规划主体改成：
  - 阶段 1：底盘定位
  - 阶段 2：升降杆到目标高度附近
  - 阶段 3：只用右臂求“预备姿势”
  - 阶段 4：只用右臂完成短距离吸取
  - 阶段 5：只用右臂回撤


## 1. 新的六阶段定义

建议把状态机固定成下面 6 段。

### 1. `OBSERVE`

输入：

- 目标位姿 `target_pose`
- 当前机器人状态
- 当前环境点云/障碍信息

输出：

- 目标法向
- 目标高度
- 目标在货架中的层位
- 目标附近可进入方向

这一段不做轨迹规划，只做任务几何计算。


### 2. `STOW_SAFE`

目的：

- 把右臂收回到安全弯曲区间
- 如果升降杆当前高度危险，先把升降杆移动到一个安全中间高度

特点：

- 升降杆单独动作
- 右臂单独动作
- 不再做联合轨迹


### 3. `BASE_ALIGN`

目的：

- 底盘运动到适合抓取的局部站位
- 朝向目标，但不一定让底盘中心正对目标(注意目前使用的simple3这个urdf,90才是机器人正对的方向)

输出：

- `base_x`
- `base_y`
- `base_theta`

特点：

- 这是底盘自己的局部位姿求解，不属于 MoveIt 联合轨迹
- 可以由导航/底盘模块执行


### 4. `LIFT_ALIGN`

目的：

- 升降杆到目标高度附近
- 不追求精确到最终吸取高度，只要求给右臂留出合适的工作区

输出：

- `raise_joint_target_height`

特点：

- 升降杆单独执行
- 到位后再进入下一段


### 5. `PREPARE_ARM`

目的：

- 计算一个“动态预备姿势”
- 这个姿势不是写死关节值，而是由目标位姿、当前底盘位姿、当前升降杆高度共同决定

标准：

- 吸盘姿态与目标姿态一致
- 吸盘与目标在“垂直于吸取方向的平面”上的投影重合
- 吸盘距离目标还有一个准备距离 `d_prepare`
- 大臂与小臂保持在一个合适的弯曲角度范围
- 该姿势下最终存在一条“只用右臂短距离推进”的可行解

这一段是新版算法的核心。


### 6. `FINAL_APPROACH_AND_RETREAT`

可以在状态机里逻辑上拆成两个子阶段：

- `FINAL_APPROACH`
- `RETREAT`

但从规划策略上它们很像：

- 都只允许右臂参与
- 都保持末端姿态基本不变
- 都沿目标法向做短距离直线运动

`FINAL_APPROACH`：

- 从预备姿势沿吸取方向直线插入

`RETREAT`：

- 吸取后沿反方向退出，再视需要上抬一点


## 2. 为什么旧方案不适合真机

旧方案的问题不是“某个参数错了”，而是模型假设错了：

1. 升降杆被当成连续轨迹轴
2. 预备姿势有时被当成固定姿态
3. 底盘、升降杆、右臂的职责没有真正解耦

这会导致：

- 仿真里能规划，真机不能按时序执行
- 准备姿势不随目标和初始状态变化
- 最终吸取段依赖错误的联合自由度


## 2.1 旧版的严重动作冗余问题

除了“升降杆被错误当成连续规划轴”之外，旧版还有一个非常严重的问题：

- 它经常生成很多和任务目标无关的冗余动作

典型现象包括：

- `r3` 这种 roll 轴为了到达同一个末端姿态，突然额外转接近 `180°`
- 升降杆先上去再下来，或者先下来又上去
- 手臂先做一次预备动作，后面又重复做一次相似的预备动作
- 明明已经处在可插入姿态附近，却还先绕一圈再去吸取

这些动作在仿真里可能还能跑，但对真机非常不友好，因为它会带来：

- 总动作时间明显变长
- 轨迹不稳定、不可解释，难以调试
- 关节多余翻转，增加碰撞和缠绕风险
- 升降杆无意义反复动作，降低执行效率
- 视觉上“像在乱动”，不符合真实项目对可控性的要求

所以新版方案除了要解决“离散升降杆”的问题，还必须把“冗余动作最小化”作为硬约束，而不是优化项。

后续实现时应该明确遵守这些原则：

- 升降杆每个任务周期最多只做少量阶段性动作，不允许无意义往返
- `r3` 这类 roll 轴只用于修正前臂/吸盘朝向，不允许为了数学上等价的姿态产生大幅翻转
- `PREPARE_ARM` 必须直接服务于 `FINAL_APPROACH`，不允许连续生成两次本质相同的预备动作
- 如果当前已经接近目标可插入姿态，应优先走短路径，而不是重新绕回模板姿态
- 候选姿态评分里要加入“与当前状态变化量最小”和“关节翻转惩罚”

另外，最近排查又确认了一个非常关键的根因：

- 如果初始化时右臂已经落在错误的弯曲分支上，尤其是 `r4` 一开始就是反向弯曲，
- 后面的 `stow -> prepare -> final` 很容易一路沿着这个错误分支继续求解，
- 即使数值上还能解出 IK，动作也会一直带着“反向补偿”的倾向。

所以新版除了改阶段逻辑，还必须把“初始化分支一致性”列为硬约束：

- 默认 `home/ready/stow` 不能继续使用旧的错误分支姿态
- 随机初始化不能独立随机每个关节，而要围绕几组同一弯曲分支的模板采样
- `PREPARE_ARM` 的 seed 应尽量和初始化/收臂阶段保持同一类弯曲分支


## 3. 新版预备姿势的定义

预备姿势 `prepare_pose` 不再是固定关节角，而是一个由目标几何推出来的姿态。

定义如下：

- 设目标位姿为 `T_target`
- 设目标吸取方向单位向量为 `n`
- 设准备距离为 `d_prepare > 0`

则预备姿态满足：

- `R_prepare = R_target`
- `p_prepare = p_target - d_prepare * n`

也就是：

- 预备姿势和目标姿势朝向相同
- 位置只在法向上后退了一小段
- 在垂直于法向的平面内，二者投影重合

这正是你要求的那种：

- “在目标位姿向量垂直的那个平面上的投影坐标相同”


## 4. 动态预备姿势不是直接求 joint，而是分两层求解

这一步不能简单写成“对 `prepare_pose` 做一次 IK”。

应该拆成两层：

### 第一层：几何层

根据目标位姿计算：

- 目标法向 `n`
- 预备姿势位置 `p_prepare`
- 目标层高 `z_target`
- 推荐升降杆高度 `h_lift`
- 推荐底盘站位 `(x_base, y_base, theta_base)`

### 第二层：右臂可达性层

在底盘和升降杆已经确定后，检查右臂是否能满足：

- 姿态正确
- 弯曲度合理
- 与货架不碰撞
- 最终还能继续前推到 `target_pose`

如果不满足，就回到第一层调底盘站位和升降杆高度，再重试。


## 5. 动态预备姿势的判断标准

后面真正写代码时，`prepare_pose` 不应是“第一个 IK 解就用”，而要做筛选。

建议使用下面这些标准：

### 必须满足

- 吸盘姿态和目标姿态误差在阈值内
- 吸盘与目标的横向投影误差足够小
- 不碰撞
- 最终插入段可行

### 优先更好的解

- `raise_joint` 更接近中间工作区，不靠上下极限
- `r2/r4` 落在一个能形成前伸弯曲的区间内
- `r3` 落在一个合适的滚转区间内，使小臂/吸盘朝向货架开口
- 末端离货架入口距离合理，不过深也不过远
- 从当前手臂状态移动到预备姿势代价更小

可以把它写成一个评分函数：

```text
score = 
  w1 * 姿态误差 +
  w2 * 横向投影误差 +
  w3 * 弯曲度惩罚 +
  w4 * 与关节中值距离 +
  w5 * 与当前姿态变化量 +
  w6 * 与障碍最小距离惩罚
```

分数最低的那个候选预备姿势才采用。


## 6. 什么叫“手臂弯曲”

这件事不要再写成固定关节值，而应该写成关节约束区间。

比如可以定义：

- `r2_joint` 在一个抬臂区间内
- `r4_joint` 在一个肘部弯曲区间内
- `r3_joint` 在一个允许的小臂滚转区间内

更实用的是定义“肘部弯曲度”：

- 避免上臂和前臂几乎共线
- 避免手臂完全折死
- 这个量不应直接用 `r3` 判断，因为 `r3` 是 roll 轴，主要影响小臂伸出去时的朝向

例如：

```text
elbow_bend_score = f(r2_joint, r4_joint, 上臂方向, 小臂方向)
```

然后要求它落在一个可用区间：

```text
elbow_bend_min <= elbow_bend_score <= elbow_bend_max
```

这样就比“写死一组 joint”更能适应目标高度和初始位姿变化。


## 7. 新版目标求解流程

建议后续实现成下面这个顺序。

### 步骤 A：输入目标位姿

视觉模块输出：

- `target_pose`

### 步骤 B：几何计算

从 `target_pose` 算：

- 吸取方向 `n`
- 预备姿势 `prepare_pose`
- 建议底盘站位候选集合
- 建议升降杆高度候选集合

### 步骤 C：搜索 `(base, lift)` 组合

例如：

- 底盘前后偏移若干候选
- 底盘横向偏移若干候选
- 底盘朝向若干候选
- 升降杆高度若干候选

对每个组合：

1. 固定底盘和升降杆
2. 用右臂去解 `prepare_pose`
3. 检查碰撞
4. 检查弯曲度
5. 检查是否存在 `prepare -> target` 的短距离只用右臂插入解

找到第一个好解，或者选评分最低的最佳解。

### 步骤 D：执行状态机

1. `STOW_SAFE`
2. `BASE_ALIGN`
3. `LIFT_ALIGN`
4. `PREPARE_ARM`
5. `FINAL_APPROACH`
6. `RETREAT`

### 右臂接口

- 继续由 MoveIt + controller 执行


## 9. 下一步代码应该怎么改

后面不要继续在当前状态机上“打一块补丁”，而是按下面顺序重构：

1. 新增一个几何求解模块  
   负责从 `target_pose` 生成：
   - `base_goal`
   - `lift_target_height`
   - `prepare_pose`
   - 候选评分

2. 在 `pick_task_manager.py` 里把 `OBSERVE` 改成“求候选解”  
   而不是直接生成固定准备姿态

3. 去掉“准备姿态固定 joint”这一逻辑  
   保留它只作为初始 seed，不作为最终答案

4. `FINAL_APPROACH` 保持只用右臂笛卡尔插入

5. `RETREAT` 保持只用右臂回撤


## 10. 当前最重要的实现目标

如果只抓重点，下一版应该先做到这三点：

1. 底盘和升降杆从联合规划里彻底剥离  
2. 预备姿势从“固定姿态”改成“动态求解 + 评分筛选”  
3. 最终吸取段保证只用右臂也能完成  

这三点成立以后，这套算法才是真正接近真机部署逻辑的。


## 11. 新版进展可视化

为了边改边看新版候选搜索的效果，现在可以直接运行：

```bash
bash scripts/run_moveit_mock_progress.sh
```

这个入口会打开：

- MoveIt RViz
- 机器人模型
- 货架点云
- 红色最终目标位姿
- 绿色实际执行末端位姿
- 橙色阶段路径
- 新版候选搜索的调试标记

其中调试标记的含义是：

- 白色文字：当前状态机阶段
- 红色球/箭头：被拒绝的底盘候选站位
- 黄色球/箭头：通过基本可行性检查的候选
- 绿色球/箭头：当前选中的候选

这样可以直接看到：

- 新版有没有在搜索候选
- 候选主要分布在哪些站位
- 当前为什么会回退
- 选中的准备解是否直观合理
