# 相机构图后如何去掉贴地低点

这份文档写的是一个很实际的问题：

- 相机或者建图模块已经给了你一张整体点云图
- 但是地面附近有很多点
- 这些点会让 MoveIt 误以为机器人前面、底盘周围、货架前方到处都是障碍
- 结果就是规划变差，甚至直接规划失败

如果你现在的场景里，真正有用的目标和货架内容都在离地 `20 cm` 以上，
并且货架本体属于你已知的静态结构，那么最直接的办法就是：

- 保留已知货架结构点
- 再把其它 `z < 0.20 m` 的点过滤掉

这不是最“智能”的办法，但在很多货架取药场景里非常实用，而且最稳定。

## 1. 为什么贴地低点会干扰规划

MoveIt 不知道哪些点是“没用的地面噪声”，哪些点是真正障碍。

所以只要这些点还在点云里：

- octomap 会把它们体素化成障碍
- 机械臂靠近货架前方时会被误判碰撞
- 底盘附近也会被堆出一层假障碍

常见来源有：

- 地面反光
- 深度相机边缘噪声
- 建图时地面残留
- 轮子、底座、线缆附近的杂点

## 2. 最简单有效的方法：高度阈值过滤

前提是：

- 你的点云已经在 `world` 或者某个稳定世界坐标系下
- `z` 轴朝上
- 你知道有用物体不会低于某个高度

这时最简单的实现是：

```text
货架结构点直接保留
非货架点里：
    保留 z >= 0.20 m 的点
    删除 z < 0.20 m 的点
```

这样就不会把靠地面的货架立柱、背板下沿一起误删掉。

## 3. 什么时候 20 cm 合适

如果你的场景满足下面条件，`0.20 m` 很合适：

- 货架最低层板本身就高于地面
- 药盒不会直接摆在地上
- 机器人需要关注的是货架中上部，不是地面拾取

如果以后你要抓更低的物体，可以再调小，比如：

- `0.10`
- `0.05`

但在你当前这个货架场景里，先用 `0.20` 更稳。

## 4. 这一步应该放在整条链的哪里

推荐顺序是：

```text
相机/建图输出整体点云
-> 坐标变换到 world
-> 去掉离群点
-> 去掉贴地低点
-> 需要的话再做体素降采样
-> 再送给 MoveIt / octomap
```

不要把“原始未处理点云”直接送进 MoveIt。

## 5. 你当前工作区里已经怎么做了

我已经把当前 mock 环境点云默认改成：

- 货架结构点全部保留
- 非货架点里，`z < 0.20 m` 的点直接过滤掉

相关代码在：

- [mock_vision_publisher.py](/home/lzoolloozl/projects/medipick_moveit_workspace/src/medipick_planning_server/scripts/mock_vision_publisher.py)
- [moveit_mock_demo.launch.py](/home/lzoolloozl/projects/medipick_moveit_workspace/src/medipick_planning_server/launch/moveit_mock_demo.launch.py)

其中参数名是：

- `min_point_z`

默认值现在就是：

```text
0.20
```

## 6. 现在怎么启动

直接启动：

```bash
bash scripts/run_moveit_mock_demo.sh rviz:=true
```

这时 mock 点云环境已经会默认过滤掉 `20 cm` 以下的点。

如果你想自己改阈值，也可以：

```bash
bash scripts/run_moveit_mock_demo.sh rviz:=true min_point_z:=0.15
```

或者：

```bash
bash scripts/run_moveit_mock_demo.sh rviz:=true min_point_z:=0.25
```

## 7. 如果以后换成真实相机点云，应该怎么做

真实点云不要依赖 mock 节点，而是自己做一个“点云预处理节点”。

这个节点至少做三件事：

1. 接收原始 `PointCloud2`
2. 如果你有已知静态结构模型，先把这些结构点保留
3. 再过滤掉非结构点里 `z < 0.20 m` 的点
4. 把过滤后的点云重新发布给 MoveIt 使用

最小逻辑其实很简单：

```text
for point in cloud:
    if point_belongs_to_known_shelf_structure(point):
        keep
    elif point.z >= 0.20:
        keep
    else:
        drop
```

然后把新点云发到一个新话题，例如：

- `/medipick/filtered_pointcloud`

再在 MoveIt 启动参数里把点云输入话题改成这个过滤后的话题。

## 8. 真实项目里推荐再加两步

只做高度阈值已经够你现在先跑规划了。

但如果后面想更稳定，建议再加：

- 体素降采样
  减少点数，降低 octomap 更新负担

- 平面分割
  用 RANSAC 去掉真正的地面平面，而不是只靠固定高度

这样适应性会更强。

## 9. 什么时候不要只用高度阈值

下面这些情况，不适合只靠“全局删掉 `z < 0.20`”：

- 相机坐标系还没对齐世界坐标系
- 地面不是水平的
- 目标物本来就放在很低位置
- 机器人以后要做地面拾取

这时候更适合：

- 先做 TF 对齐
- 再做平面分割
- 再做区域裁剪

## 10. 给你现在的建议

你现在的目标是先验证：

- 多层货架环境
- 整体点云进入 MoveIt
- 在这个环境里规划和执行

所以当前最推荐的策略就是：

- 先固定用 `min_point_z = 0.20`
- 先把地面低点去掉
- 先把规划链跑顺

等这一步稳定了，再考虑更复杂的点云分割和建模。
