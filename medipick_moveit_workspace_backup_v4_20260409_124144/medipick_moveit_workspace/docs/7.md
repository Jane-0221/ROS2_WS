# 当前版本用法与项目详解

## 1. 当前版本是什么

这是 `v2` 阶段的工作版本，重点不再是单纯演示，而是：

- 用 `pick_task_manager.py` 驱动拾取状态机
- 用 `planning_server.py + MoveIt` 做规划
- 在 mock 货架点云环境里验证
  - 动态 `prepare_pose` 候选搜索
  - `final_approach` 的局部插入
  - 离散升降杆模式

当前版本的源码备份已经放在父目录：

- `/home/lzoolloozl/projects/medipick_moveit_workspace_backup_v2.tar.gz`

## 2. 当前重点解决的问题

这一版主要围绕以下问题修复：

- 升降杆不能再和手臂、底盘一起做连续联合轨迹
- `prepare_pose` 不能写死，必须根据目标、底盘、升降杆高度动态选
- `final_approach` 之前经常出现
  - 末端去的不是红色目标
  - `r2/r3/r6` 大幅绕圈
  - 自碰撞或碰到头部
  - 明显多余动作

当前修复思路是：

- 升降杆离散化
- `PREPARE_POSE / FINAL_APPROACH / RETREAT` 在离散模式下统一使用 `arm_no_lift`
- `prepare_candidate` 先搜索，再用最终阶段可行性反向筛掉差候选
- `final_approach` 优先尝试“以准备姿态为 seed 的近解 IK + 局部状态有效性验证”

## 5. 当前版本里最重要的参数

### 5.1 `prepare_offset`

含义：

- `prepare_pose = target_pose` 沿吸取方向往外退多少

经验：

- 太大：最后一段需要伸太多，容易出现大关节变化
- 太小：预备姿态本身更难解，或过早接近货架

当前测试中更推荐：

- `0.06`

最近固定场景回归里，`0.06` 比 `0.10` 更容易选出合适的动态 `prepare_candidate`，
并减少 `final_approach` 再次大幅补偿腕部/前臂关节的情况。

### 5.2 `final_prefer_seeded_ik`

含义：

- `final_approach` 是否优先使用“以准备姿态为 seed 的近解 IK”

当前建议：

- 开启

因为最后一段应该是局部插入，不应该突然跳到另一支 IK 解。

### 5.3 `task_stage_timeout`

含义：

- 每个阶段允许的最长执行/搜索时间

说明：

- 旧版默认 20 秒
- 新版候选搜索更复杂，mock 测试里已放宽到 60 秒，否则容易出现“候选还没搜完，阶段先超时”

## 6. 当前版本的已知结论

### 6.1 已确认比旧版好的地方

- 离散升降杆逻辑已经接上
- 候选 `prepare_pose` 不再完全写死
- `final_approach` 已经支持优先走“近解 IK + 局部验证”
- 当前固定场景下，已经重新出现“动态选中 candidate 且整条链完成”的结果

### 6.2 仍然没彻底收完的问题

- `shape_mask` 相关 transform warning 还在
- 不同随机场景下，动态候选并不是每次都能选出来
- `final_approach` 虽然已经比之前好，但还要继续压缩冗余动作
- 还需要继续减少
  - wrist roll 大幅翻转
  - 多余的预备摆动
  - 和头部/自身过近的轨迹

## 7. 这版建议怎么继续做

下一步最值得继续收的，是这三件事：

1. 继续优化 `prepare_candidate` 的几何生成

- 让候选更贴近“同高、同投影、只差法向插入”

2. 继续优化 `final_approach`

- 尽量让它稳定走“近解 IK 局部插入”
- 减少 fallback 到旧笛卡尔或普通 OMPL 规划

3. 把当前有效的固定场景回归，扩展到更多随机场景

- 随机货架层高
- 随机目标层
- 随机初始底盘姿态
- 随机初始手臂姿态

## 8. 交给别人时最该说明的事

如果把这版交给别人继续做，最重要的不是“从头理解整个仓库”，而是先理解这三条：

1. 升降杆已经被当成离散轴，而不是连续联合规划轴
2. `prepare_pose` 的好坏，直接决定 `final_approach` 是否会出现绕圈和自碰撞
3. 当前最关键的优化目标不是“再换更多 planner”，而是让
   - `prepare_candidate`
   - `final_approach local IK`
   这两段更一致
